# Document Format Audit Skill 使用说明（中文）

本文档面向 DeerFlow 项目维护者和使用者，详细说明 `skills/custom/document-format-audit` 这个 Skill 的用途、能力边界、执行流程、脚本参数与输出格式，以及如何将审查结果转成可交付的审核报告。

## 1. Skill 定位

`document-format-audit` 是一个“文档格式合规审核”技能，目标是把文档审核从“纯主观人工检查”变成“可重复、可解释、可追踪”的流程。

它的核心思路是：

1. 先做确定性结构检查（脚本可稳定识别的项）
2. 再做语言质量审阅（语义层面由模型或人工补充）
3. 最后输出结构化审核报告（带证据、建议、优先级）

对应文件：

- Skill 定义：[SKILL.md](/data/ht/deer-flow/skills/custom/document-format-audit/SKILL.md)
- 自动检查脚本：[audit_docx.py](/data/ht/deer-flow/skills/custom/document-format-audit/scripts/audit_docx.py)
- 默认检查清单：[checklist.md](/data/ht/deer-flow/skills/custom/document-format-audit/references/checklist.md)
- 报告模板：[report-template.md](/data/ht/deer-flow/skills/custom/document-format-audit/references/report-template.md)

## 2. 适用场景

当用户上传 `.docx`、`.doc`、`.pdf` 并提出以下诉求时，应该启用该 Skill：

- “审核格式是否规范”
- “检查排版和标题层级”
- “检查目录、页码、页眉页脚”
- “检查表格是否有错乱或越界风险”
- “检查错别字、语法、标点、术语一致性”

典型业务场景：

- 项目申报书、方案文档、制度规范、研究报告的交付前质检
- 招投标材料、政府/企业公文模板一致性检查
- 多人协作写作后的一致性回归检查

## 3. 能力边界（务必先看）

### 3.1 支持强度

- `.docx`：支持较完整的结构级自动审核（推荐输入格式）
- `.doc`、`.pdf`：仅支持基础项 + 明确标注 `manual`（人工复核）

### 3.2 机器可稳定判断 vs. 不可稳定判断

可稳定判断（确定性）示例：

- 文件命名是否匹配正则规则
- 文件扩展名是否在允许集合内
- 页面尺寸是否接近 A4（twips 容差）
- 页边距是否落在常见范围
- 是否检测到 TOC / PAGE 字段
- 标题层级是否有明显跳级
- 正文主导字体/字号/行距是否一致
- 表格列结构是否异常、是否存在溢出风险

不可稳定完全自动判断（需人工或模型补充）示例：

- `.doc` / `.pdf` 的精确版式核查
- 图表视觉越界、对象遮挡、分页断裂等渲染级问题
- 语义类错别字、语病、上下文逻辑和术语规范

结论：这个 Skill 不是“百分之百替代人工”，而是“先机器筛查，再人工/模型补充”。

## 5. 标准执行流程

Skill 的执行流程建议固定为 4 步，保证结果可复现。

### 第 1 步：确认审核范围和标准

- 确认目标文件路径（通常位于 `/mnt/user-data/uploads/`）
- 确认是否有“单位自定义标准”：
  - 字体/字号
  - 页边距
  - 标题层级规则
  - 文件命名规则
- 如果用户未提供标准，使用默认 `references/checklist.md`

### 第 2 步：运行确定性脚本检查

示例（默认规则）：

```bash
python /mnt/skills/custom/document-format-audit/scripts/audit_docx.py \
  --file /mnt/user-data/uploads/<filename>.docx \
  --output-json /mnt/user-data/outputs/<filename>.audit.json
```

示例（自定义命名规则）：

```bash
python /mnt/skills/custom/document-format-audit/scripts/audit_docx.py \
  --file /mnt/user-data/uploads/<filename>.docx \
  --filename-rule '^[A-Za-z0-9_\\-\\u4e00-\\u9fa5]+-[A-Za-z0-9_\\-\\u4e00-\\u9fa5]+-(\\d{8}|v\\d+(\\.\\d+)*)\\.(docx|doc|pdf)$' \
  --output-json /mnt/user-data/outputs/<filename>.audit.json
```

本地开发环境也可直接调用仓库内脚本路径：

```bash
python skills/custom/document-format-audit/scripts/audit_docx.py \
  --file /path/to/your.docx \
  --output-json /tmp/your.audit.json
```

### 第 3 步：补充语言质量审阅

建议优先使用已转换 Markdown（如 `/mnt/user-data/uploads/<filename>.md`）进行语义审核，重点关注：

- 错别字与病句
- 标点和格式符号
- 术语前后是否统一
- 表述是否冗长或歧义

### 第 4 步：输出审核报告

按 `references/report-template.md` 汇总：

- 每条发现写清楚 `status`、`evidence`、`suggestion`、`priority`
- 生成最终结论：`Pass / Conditional pass / Fail`

## 6. 审核项与脚本能力映射

下表可直接回答“这个 Skill 能不能做某项检查”。

| 审核需求 | 脚本检查项（item） | 自动化程度 | 说明 |
|---|---|---|---|
| 文件命名规范 | `File naming convention` | 自动 | 基于 `--filename-rule` 正则 |
| 文件类型正确性 | `Allowed file type` | 自动 | 允许 `docx/doc/pdf` |
| 页面尺寸（A4） | `A4 page size` | 自动（docx） | 使用 twips 与 A4 容差比较 |
| 页边距 | `Margins in typical range` | 自动（docx） | 默认常见范围 900-2000 twips |
| 页面方向 | 含在 `A4 page size` 证据中 | 自动（docx） | 从 `pgSz` 和宽高关系推断 |
| 目录完整性 | `TOC presence` | 自动（docx） | 检测 XML 中 `TOC` 字段 |
| 页眉页脚 | `Header/Footer presence` | 自动（docx） | 检测引用关系 |
| 页码 | `Page number field` | 自动（docx） | 检测 `PAGE` 字段 |
| 标题层级连续性 | `Heading hierarchy continuity` | 自动（docx） | 识别 `Heading1/标题1` 并找跳级 |
| 字体一致性 | `Font consistency` | 自动（docx） | 按主导比例判定，一般阈值 0.7 |
| 字号一致性 | `Font size consistency` | 自动（docx） | 按主导比例判定，一般阈值 0.7 |
| 行距一致性 | `Line spacing consistency` / `Line spacing detectability` | 半自动（docx） | 无显式行距时记为 `manual` |
| 首行缩进 | `First-line indent usage` | 自动（docx） | 检测段落 `firstLine` 属性 |
| 表格结构 | `Table structure consistency` | 自动（docx） | 按行列数一致性检测 |
| 表格越界风险 | `Table overflow risk` | 自动（docx） | 估算表宽与可写宽度 |
| 标点重复 | `Repeated punctuation` | 自动（docx） | 规则匹配 |
| 括号/引号配对 | `Bracket/quote balance` | 自动（docx） | 左右计数不平衡告警 |
| 错别字/语法 | `Typos and grammar` | 手工/模型 | 脚本明确给 `manual` |
| 图表越界/错乱 | 无固定自动项 | 手工为主 | 建议人工视觉复核 |

## 7. `audit_docx.py` 参数说明

`audit_docx.py` 是该 Skill 的核心执行器，参数如下：

- `--file`（必填）：目标文件路径，支持 `.docx/.doc/.pdf`
- `--filename-rule`（可选）：文件名合规正则，默认值见脚本 `DEFAULT_FILENAME_RULE`
- `--output-json`（可选）：输出 JSON 结果文件路径

默认命名规则：

```regex
^[A-Za-z0-9_\-\u4e00-\u9fa5]+-[A-Za-z0-9_\-\u4e00-\u9fa5]+-(\d{8}|v\d+(\.\d+)*)\.(docx|doc|pdf)$
```

含义：

- 第一段：项目名
- 第二段：单位名
- 第三段：日期（`YYYYMMDD`）或版本（如 `v1`, `v1.2`）
- 后缀：`docx/doc/pdf`

## 8. 输出 JSON 结构解读

脚本输出 JSON 的核心结构如下：

```json
{
  "file": "...",
  "filename": "...",
  "file_extension": "docx",
  "generated_at": "...",
  "checks": [
    {
      "item": "A4 page size",
      "status": "pass",
      "severity": "info",
      "evidence": "...",
      "suggestion": "..."
    }
  ],
  "summary": {
    "pass": 0,
    "fail": 0,
    "warn": 0,
    "manual": 0,
    "total": 0
  },
  "signals": {
    "page": {},
    "headings": {},
    "typography": {},
    "tables": {},
    "language_signals": {}
  }
}
```

字段语义：

- `checks`：逐项审核结果，供报告直接引用
- `status`：`pass | fail | warn | manual`
- `severity`：`high | medium | low | info`
- `summary`：总计数，适合做看板或阈值决策
- `signals`：原始信号数据，便于深挖和复核

### `signals` 重点字段

- `signals.page`：页面宽高、方向、页边距（含 cm 转换）
- `signals.headings.counts_by_level`：各级标题数量统计
- `signals.typography`：主导字体/字号/行距比例、首行缩进采样
- `signals.tables`：表格总数、结构异常数、越界风险数
- `signals.language_signals`：重复标点、多空格、不平衡符号计数

## 9. 状态与结论判定建议

该 Skill 输出的是“事实 + 信号”，最终业务结论建议按以下规则归并：

- `Pass`
  - 无 `high` 严重问题
  - `fail` 为 0
  - 仅少量低风险 `warn`
- `Conditional pass`
  - 无阻断性问题
  - 存在需要修订的 `warn` 或 `manual` 项
- `Fail`
  - 出现高严重级别 `fail`
  - 或关键合规项不满足（例如必须 A4/必须页码但缺失）

建议在报告中保留“修复优先级队列”：

- `High`：影响交付或合规
- `Medium`：建议发版前修复
- `Low`：风格优化项

4. 输出结论与修订清单

## 11. 常见问题（FAQ）

### Q1：为什么 `.doc` / `.pdf` 会出现很多 `manual`？

因为当前脚本基于 `.docx` 的 OpenXML 结构解析，`.doc` 和 `.pdf` 无法直接提供同等结构信号，所以会明确降级为人工复核项。

### Q2：行距为什么有时显示 `Line spacing detectability`？

部分文档没有写入显式行距属性，而是依赖样式继承或编辑器默认值。此时脚本不会猜测，按 `manual` 处理更稳妥。

### Q3：表格“越界风险”是不是一定越界？

不是。该项是基于宽度估算的风险提示，遇到合并单元格、自动适应、图片嵌入等复杂布局时，仍需人工视觉确认。

### Q4：语言质量能否全自动？

目前不建议。脚本只做“信号检测”，语义层面的错别字、语病、术语一致性仍应由模型或人工二次审核。

## 12. 扩展建议

若要把该 Skill 从“可用”提升到“生产级严格审校”，可以按以下方向增强：

- 增强 `.pdf` 版面解析能力（结合渲染坐标）
- 引入模板规则配置化（按组织/部门加载不同标准）
- 增加图表、图片、分页断裂等视觉检测
- 增加术语库、禁用词库、标点风格库
- 把 `summary` 接入质量门禁（CI 或交付前自动阻断）

## 13. 维护建议

- 修改检查标准时，先更新 `references/checklist.md`
- 修改报告结构时，同步更新 `references/report-template.md`
- 扩展脚本检查项时，保持 `checks` 字段语义稳定，避免下游解析失效
- 对外说明时，始终明确“自动检查范围”和“manual 范围”
